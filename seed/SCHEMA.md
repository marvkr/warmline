# Warmline — Demo Data Schema (the contract)

> `convex/schema.ts` currently defines only `authTables` + a stub `numbers` table — there is **no
> domain schema yet**. So this file IS the contract: it documents every entity, field, and the
> confidence method, in the shape of PHASE_0 §5, so Marvin can wire the Convex backend fast. Field
> names are a starting point — rename freely; the dataset (`seed/demo-data.json`) is regenerated by
> `seed/generate.mjs` and can be reshaped to whatever the backend finalizes.

The dataset is a single JSON document, `seed/demo-data.json`, with top-level keys:
`_meta`, `flags`, `goals`, `people`, `events`, `edges`, `recommendations`.

---

## Entities

### Person  (`people[]`)
| field | type | notes |
|---|---|---|
| `id` | string | stable slug, e.g. `"han"` |
| `name` | string | public name |
| `headline` | string | public role/title |
| `company` | string | public company |
| `linkedin_url` | string \| null | public profile URL; `null` if none in source |
| `x_handle` | string \| null | handle without `@`; `null` if none |
| `avatar_url` | string \| null | left `null` — backend/UI supplies avatars |
| `is_self` | bool | the protagonist (consenting account). Exactly one `true`. |
| `is_target` | bool | `true` for the 12 hero fan-out targets |
| `roles` | string[] | tags: `gatekeeper` / `fanout` / `judge` / `go_cold` / `support` / `self` |
| `source` | string | provenance: `linkedin_connection` / `luma_gala` / `luma_convex_hh` |

### Edge  (`edges[]`)
| field | type | notes |
|---|---|---|
| `from_id` | string | Person.id |
| `to_id` | string | Person.id |
| `type` | `"linkedin_1st"` \| `"co_attended_event"` | |
| `confidence` | number 0–1 | for `linkedin_1st`: certainty the real connection exists (0.95). for `co_attended_event`: **model-generated reachability score** (see method below). |
| `evidence` | string | human-readable, traces to a real export. e.g. `"Both attended Mintlify Gala, 2026-04-16"` or `"LinkedIn 1st-degree connection since 2025-08-16"` |
| `confidence_tier` | `"High"` \| `"Medium"` \| `null` | tier for co-attendance edges; `null` for linkedin_1st |
| `confidence_generated` | bool | `true` when `confidence` is a model score (Claims Ledger §6) |
| `judge_kicker` | bool | `true` ONLY for the optional judge edges (Han→Danylo, Purav→Wayne) |
| `event_id` | string \| null | the event backing a `co_attended_event` edge |

### Event  (`events[]`)
| field | type | notes |
|---|---|---|
| `id` | string | slug |
| `name` | string | real Luma event name |
| `date` | string | ISO `YYYY-MM-DD` |
| `attendee_ids` | string[] | Person.ids present (real attendance) |

### Recommendation  (`recommendations[]`)
| field | type | notes |
|---|---|---|
| `id` | string | slug |
| `type` | `"gatekeeper"` \| `"go_cold"` | which beat it drives |
| `person_id` | string | who to act on |
| `goal_id` | string | the goal it serves |
| `score` | number 0–1 | drives ranking |
| `reason_bullets` | string[] | why-them / why-now |
| `trigger` | string \| null | go-cold "why now" signal (model-generated; see Claims) |
| `how` | object | `{ channel, angle, drafted_opener }` — `drafted_opener` is LLM-generated |
| `how_generated` | bool | `true` (openers + angles are illustrative per §6) |
| `unlocks_ids` | string[] | the gatekeeper fan-out this rec opens |

### Goal  (`goals[]`)
| field | type | notes |
|---|---|---|
| `id` | string | slug |
| `text` | string | e.g. `"Break into SF dev tools"` |
| `goal_vector` | null | placeholder for backend embedding-based reranking |

### Flags  (`flags`)
- `include_judge_edges` (bool) — the **clearly-named toggle**. When a consumer renders with this
  `false`, it must drop every Person with role `judge` and every Edge with `judge_kicker === true`.
  The hero path + 12 fan-out remain fully intact without them.

## Confidence method (explainable, deterministic, NEVER randomized)

Applied to `co_attended_event` edges (the model's "how reachable is this person via the gatekeeper").
Implemented verbatim in `seed/generate.mjs` (`scoreReachability()`):

```
c = 0.45                              # base: verified co-attendance with the gatekeeper at the Gala
  + 0.28  if target is also a direct 1st-degree LinkedIn connection of self   # independent warm path
  + 0.25  if target shares the gatekeeper's employer (Mintlify)               # Han can intro internally
  + relevance_to_goal:                # deterministic, from public role/company vs "Break into SF dev tools"
        core     = 0.10   (dev-tools builder: Chroma, Cartesia, React Scan, Manifold, Letta, Expo, Mintlify)
        adjacent = 0.06   (infra media / distribution / automation: SemiAnalysis, Ben's Bites, automation edu)
        broader  = 0.03   (AI research / other: Berkeley research, OpenAI research)
c = min(c, 0.92)                      # cap — we never imply certainty about private relationships
tier = "High" if c >= 0.70 else "Medium"
```

Result (no randomness, no uniformity):
- **High:** Justin Torre 0.92 (Mintlify + connection), Gabriel Petersson 0.76 (connection + research).
- **Medium:** core builders 0.55, adjacent 0.51, broader 0.48.

`linkedin_1st` edges use `confidence = 0.95` — this is not a model guess but the certainty that a
real, scraped 1st-degree connection exists; the real connection date lives in `evidence`.

## Claims ledger mapping (what's real vs generated) — PHASE_0 §6

- **Real (say it on camera):** every Person on the hero/judge path (LinkedIn connections + Luma guest
  lists), every `evidence` string, every Event + `attendee_ids`, and the co-attendance bridges.
- **Our model (say we built it):** the `confidence` scores on `co_attended_event` edges and the live
  reranking the score enables. Flagged `confidence_generated: true`.
- **Illustrative / generated (do NOT present as ground truth):** the specific confidence *values*,
  every `drafted_opener` and `angle` (`how_generated: true`), and each go-cold `trigger`.
- **Productionization path (never "live"):** burner Luma signup, auto-sending outreach, per-user
  learning over time. Not represented as live anywhere in the data.

## PII guarantees
No emails, phone numbers, message/DM contents, or login data anywhere in `demo-data.json`. Only public
names, roles, companies, public LinkedIn URLs, public X handles, and public event attendance.
`seed/_inputs/` is gitignored and never staged. `seed/validate.mjs` regex-scans the output and FAILS
on any email/phone/DM-like pattern.
